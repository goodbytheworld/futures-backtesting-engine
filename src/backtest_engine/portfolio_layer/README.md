# Portfolio Layer

This directory contains the architecture and implementation for the Portfolio Backtest Engine, which orchestrates multiple independent trading strategies into a single synchronized portfolio context. 

The portfolio layer separates strategy execution, position tracking, capital allocation, and engine synchronization into distinct subpackages, following a robust and modular Domain-Driven Design layout.

## Subdirectories and Components

- **`adapters/`**
  - **`legacy_strategy_adapter.py`**: A compatibility wrapper that allows existing standalone `Strategy` instances (which assume they are running in the single-asset engine) to operate within the portfolio engine's synchronized environment.

- **`allocation/`**
  - **`allocator.py`**: Manages capital allocation across the active strategies in the portfolio. Responsible for sizing positions according to predefined rules and rebalancing policies based on portfolio equity.

- **`domain/`**
  - **`contracts.py`**: Defines core data structures and interfaces, such as `PortfolioConfig` (for portfolio settings) and `StrategySlot` (for individual strategy configurations within the portfolio).
  - **`policies.py`**: Houses business rules and configuration policies regarding rebalancing schedules (`RebalancePolicy`) and execution mechanics (`ExecutionPolicy`).
  - **`signals.py`**: Contains strictly defined data classes like `StrategySignal` and `TargetPosition` that represent actionable trading intents generated by the strategies.

- **`engine/`**
  - **`engine.py`**: The primary entry point (`PortfolioBacktestEngine`). It drives the master time loop, synchronizes all strategy slots to a central timeframe, and coordinates interactions between runners, allocators, and the portfolio book.

- **`execution/`**
  - **`portfolio_book.py`**: The central ledger for the portfolio. Tracks open positions, calculates real-time aggregated equity, deducts commissions, and computes bar-level cross-strategy PnL.
  - **`strategy_runner.py`**: An isolated execution environment assigned to a specific `StrategySlot`. It handles feeding data to its assigned strategy and translating the strategy's internal signals into unified `TargetPosition` objects for the engine to execute.

- **`reporting/`**
  - **`results.py`**: Handles compiling the final state of the `PortfolioBook` and execution logs into standard outcome formats. Packages history, trades, and strategy metrics for the analytics exporters, terminal UI, and shared artifact loaders to consume.

- **`scheduling/`**
  - **`scheduler.py`**: Implements temporal logic (e.g., `IntrabarScheduler`, `DailyScheduler`, `WeeklyScheduler`) to govern *when* specific actions like capital rebalancing or performance sweeps occur across the portfolio timeline.

### Configuration
- **`portfolio_config_example.yaml`**: A sample configuration file demonstrating how to mathematically describe the portfolio components, initial capital, trading pairs, and assigned strategies for an engine run.

## Output Contract

- Portfolio runs persist their canonical artifact bundle under `results/portfolio/`.
- Scenario reruns persist namespaced bundles under `results/scenarios/<scenario_id>/portfolio/`.
- These bundles are consumed by `services/artifact_service.py` and the FastAPI terminal UI in `runtime/terminal_ui/`.
